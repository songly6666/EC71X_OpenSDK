## 2 简介

本项目遵循 [Angular Commit Message Convention](https://github.com/angular/angular/blob/main/CONTRIBUTING.md#commit) 规范，
旨在通过统一的提交信息格式提高项目的可读性与可维护性。

规范化的提交信息有以下好处：

- **自动生成 CHANGELOG**：工具可根据 type 自动归类变更日志
- **快速定位变更**：通过 scope 快速筛选特定模块的历史修改
- **语义化版本管理**：根据 type 判断版本号的升级幅度（主版本 / 次版本 / 补丁）
- **提高协作效率**：团队成员无需打开代码即可理解每次提交的意图

---

## 3 提交信息格式

每条提交信息由三个部分组成：**Header**、**Body** 和 **Footer**，各部分之间用一个空行分隔。

```
（必填）<type>(<scope>): <subject>
（空行）
（可选）<body>
（空行）
（可选）<footer>
```

**示例**

```
feat(spi): 新增DMA传输模式支持

1、在SPI驱动中增加DMA传输通道配置
2、支持大数据量高速收发，降低CPU占用率
3、原有的轮询传输模式保持不变，DMA模式通过配置参数开启

Closes #45
```

> **注意**：Header 是唯一必填的部分。Body 和 Footer 可根据提交的复杂程度决定是否填写。
> 对于简单修改（如修一个拼写错误），只写 Header 即可；对于较复杂的变更，建议补充 Body 说明原因。

## 4 Header 详解（必填）

Header 是提交信息的第一行，也是最重要的部分。格式如下：

```
<type>(<scope>): <subject>
```

Header 由三个字段组成：

| 字段      | 是否必填 | 说明                     |
| --------- | -------- | ------------------------ |
| `type`    | 必填     | 提交的类型，见下表       |
| `scope`   | 可选     | 本次提交影响的模块或范围 |
| `subject` | 必填     | 简短描述本次提交做了什么 |

### 4.1 Type 类型说明

type 用于标识本次提交的性质。只允许使用以下值：

| 类型       | 说明                                                         | 版本影响   |
| ---------- | ------------------------------------------------------------ | ---------- |
| `feat`     | 新增功能或能力                                               | minor 升级 |
| `fix`      | 修复已有的 Bug                                               | patch 升级 |
| `docs`     | 仅涉及文档的新增或修改（README、注释、API 文档等）           | 无         |
| `style`    | 代码格式调整，不影响代码逻辑（空格、缩进、分号、换行等）     | 无         |
| `refactor` | 代码重构，既不是新功能也不是修复 Bug（如提取函数、重命名）   | 无         |
| `perf`     | 以提升性能为目的的代码变更                                   | patch 升级 |
| `test`     | 添加缺失的测试或修正已有测试                                 | 无         |
| `build`    | 影响构建系统或外部依赖的变更（如 Makefile、链接脚本）        | 无         |
| `ci`       | CI/CD 配置文件和脚本的变更（如 GitHub Actions、Jenkins）     | 无         |
| `chore`    | 其他杂项变更，不涉及源代码或测试文件                         | 无         |
| `revert`   | 回退某次历史提交                                             | 视情况而定 |

> **如何区分 `fix` 和 `refactor`**：如果代码修改后修复了某个已知问题或异常行为，用 `fix`；
> 如果只是改善代码结构但功能行为不变，用 `refactor`。
>
> **如何区分 `feat` 和 `perf`**：如果新增了用户可感知的能力，用 `feat`；
> 如果功能不变但运行得更快了，用 `perf`。
>
> **如何区分 `style` 和 `refactor`**：`style` 仅涉及格式层面（空格、缩进、换行），
> 代码的 AST（抽象语法树）不变；`refactor` 则涉及代码逻辑结构的调整（如提取函数、重命名变量）。

### 4.2 Scope 范围说明

scope 用于标注本次提交影响的模块或子系统，帮助在 git log 中快速筛选。
scope 应使用小写英文单词，保持简短。本项目常用的 scope 包括：

| Scope      | 适用场景                                   |
| ---------- | ------------------------------------------ |
| `gpio`     | GPIO 外设驱动相关                          |
| `i2c`      | I2C 外设驱动相关                           |
| `spi`      | SPI 外设驱动相关                           |
| `uart`     | UART 外设驱动相关                          |
| `adc`      | ADC 外设驱动相关                           |
| `pwm`      | PWM 外设驱动相关                           |
| `base`     | 底包 / 基础框架 / 系统初始化               |
| `lvgl`     | LVGL GUI 框架相关                          |
| `examples` | 示例代码                                   |
| `docs`     | 文档                                       |
| `build`    | 编译构建系统（Makefile、链接脚本、bat等）  |
| `config`   | 配置文件（引脚配置、系统参数等）           |
| `tools`    | 辅助工具及脚本                             |

> 如果本次提交涉及多个模块，可以省略 scope 或选择影响最大的模块。
> 当变更范围过大时，省略 scope 比写一个不准确的 scope 更好。

### 4.3 Subject 规则

subject 是对本次变更的一句话概括，需要遵守以下规则：

1. **长度限制**：不超过 72 个字符（Header 整行不超过 72 字符）
2. **语言**：使用中文或英文均可，但在同一项目中应保持一致
3. **不加句号**：结尾不使用句号（中文句号"。"或英文句号"."）
4. **祈使语气**：英文时使用动词原形开头（如 `add`、`fix`、`remove`），不用 `added`、`fixes`
5. **描述做了什么**：而不是描述为什么做或怎么做（那些留给 Body）

**好的 subject 示例**：

```
feat(spi): 新增DMA传输模式支持
fix(uart): 修复高波特率下分频系数溢出问题
refactor(config): 将引脚映射表抽取为独立配置数组
```

**不好的 subject 示例**：

```
feat(spi): 新增了DMA传输模式的支持。       ← 不要加句号
fix: fixed a bug                           ← 不要用过去时，且描述不够具体
update code                                ← 缺少 type，描述模糊
```

## 5 Body 详解（可选）

Body 是对 Header 中 subject 的补充说明，用于解释 **为什么做这次修改** 以及 **修改前后的行为差异**。

### 5.1 书写规则

1. 与 Header 之间必须有一个 **空行** 分隔
2. 每行不超过 72 个字符，超出时手动换行
3. 使用 **编号列表**（1、2、3、……）逐条列出变更要点
4. 重点说明 **动机（Why）** 和 **前后对比（What changed）**，而非逐行罗列改了哪些文件
5. 至少列出 2 条以上的说明条目

### 5.2 什么时候应该写 Body

- 修改的原因不是从代码 diff 中一眼能看出的
- 需要说明修改的背景或上下文
- 修改涉及性能数据的变化（建议附上前后对比数据）
- 存在替代方案但选择了当前方案（说明原因）

### 5.3 什么时候可以省略 Body

- 修改非常简单且 subject 已经足够说明（如修复一个拼写错误）
- 纯格式或文档变更

## 6 Footer 详解（可选）

Footer 用于记录两类信息：**不兼容变更** 和 **关联 Issue**。

### 6.1 不兼容变更（BREAKING CHANGE）

如果本次提交引入了与之前版本不兼容的变更（如接口签名修改、配置格式变化、移除已有功能），
**必须**在 Footer 中声明。格式如下：

```
BREAKING CHANGE: <不兼容变更的详细说明>
```

同时，建议在 Header 的 type 后面加一个 `!` 作为醒目标记：

```
feat(base)!: <subject>
```

> `BREAKING CHANGE` 必须大写，后面跟英文冒号和一个空格。
> 这是工具识别不兼容变更的标准格式，请勿使用其他写法。

### 6.2 关联 Issue

如果本次提交解决了某个 Issue，可以在 Footer 中关联：

```
Closes #123
```

支持同时关联多个 Issue：

```
Closes #123, #456, #789
```

> 使用 `Closes` 关键字后，当提交合并到主分支时，对应 Issue 会自动关闭。
> 其他支持的关键字还有：`Fixes`、`Resolves`。

## 7 完整示例

### 7.1 feat — 新功能

```
feat(spi): 新增DMA传输模式支持

1、在SPI驱动中增加DMA传输通道配置
2、支持大数据量高速收发，降低CPU占用率
3、原有的轮询传输模式保持不变，DMA模式通过配置参数开启

Closes #45
```

### 7.2 fix — Bug 修复

```
fix(uart): 修复波特率配置在115200以上时计算溢出的问题

1、原始分频系数使用uint16_t存储
2、当目标波特率超过115200时乘法中间结果溢出，导致实际波特率偏差过大
3、改用uint32_t存储中间计算结果，确保在最高921600波特率下仍然准确
```

### 7.3 docs — 文档变更

```
docs: 补充OTA升级流程说明及常见错误码列表

1、在docs目录下新增OTA章节
2、覆盖升级包制作、传输协议、校验流程
3、整理20个常见错误码及其排查方法
```

### 7.4 style — 代码格式

```
style(gpio): 统一头文件中的缩进风格为4空格

1、将gpio模块下所有头文件的制表符缩进替换为4空格缩进
2、与项目其他模块保持一致
3、不涉及任何逻辑变更
```

### 7.5 refactor — 重构

```
refactor(config): 将硬编码的引脚映射表抽取为独立配置数组

1、原先各型号的引脚映射分散在多个switch-case分支中
2、现在统一抽取到pin_map_table数组中
3、新增型号时只需在数组中添加一行配置，无需修改驱动逻辑
```

### 7.6 perf — 性能优化

```
perf(lvgl): 优化屏幕局部刷新算法减少SPI传输量

1、原有实现每次刷新都传输整屏数据（240x320）
2、现改为仅传输脏区域像素
3、在典型UI场景下，帧率从12fps提升至24fps，SPI总线占用降低约60%
```

### 7.7 test — 测试

```
test(i2c): 新增从机地址扫描的单元测试用例

1、总线上存在单个从机时正确返回地址
2、总线上存在多个从机时返回完整地址列表
3、总线空闲时返回空列表
4、SDA线被拉低时超时处理
```

### 7.8 build — 构建相关

```
build: 升级GCC交叉编译工具链至arm-none-eabi-gcc 12.3

1、旧版本10.3在开启-O2优化时存在已知的寄存器分配Bug（GCC Bugzilla #98702）
2、升级至12.3后该问题已修复
3、同步更新Makefile中的工具链路径和编译参数
```

### 7.9 chore — 杂项

```
chore: 清理gccout目录下过期的编译产物

1、删除3个月前遗留的.o和.bin文件
2、释放约120MB磁盘空间
3、该目录下的文件均为编译生成，可随时通过重新编译恢复
```

### 7.10 revert — 回退提交

```
revert: 回退 "feat(spi): 新增DMA传输模式支持"

This reverts commit a1b2c3d.

1、DMA通道与低功耗唤醒中断存在资源冲突
2、在深睡眠唤醒后DMA传输会丢失数据
3、暂时回退，待冲突解决后重新提交
```

### 7.11 BREAKING CHANGE — 不兼容变更

```
feat(base)!: 系统初始化流程改为分阶段回调机制

1、原有的单一入口函数 AppMain() 被拆分为三个阶段回调
2、AppEarlyInit()：外设时钟和引脚初始化（中断关闭状态）
3、AppInit()：驱动和中间件初始化（中断已开启）
4、AppReady()：应用逻辑启动（所有子系统就绪）

BREAKING CHANGE: AppMain() 已移除，所有应用代码需迁移至
新的三阶段回调接口。迁移指南详见 docs/migration-v2.md。
```

## 8 特殊情况处理

### 8.1 一次提交涉及多个 type

原则上 **每次提交只做一件事**。如果一次修改同时包含新功能和 Bug 修复，
应拆分为两次独立的提交。如果确实无法拆分，以 **主要变更** 的 type 为准。

### 8.2 提交信息需要修改

如果发现最近一次提交的信息有误，可以使用 `git commit --amend` 修改。
如果需要修改更早的提交，使用 `git rebase -i`（仅限未推送到远程的提交）。

> **警告**：不要对已经推送到远程的提交执行 amend 或 rebase，
> 这会导致其他团队成员的本地仓库出现冲突。

### 8.3 合并提交（Merge Commit）

合并分支时产生的 merge commit 可以保留 Git 自动生成的信息，无需遵循此规范。

## 9 建议

1. **原子性提交**：每次提交只做一件事，避免将不相关的修改混在一起
2. **先写 Header 再写代码**：在动手修改前先想好 type 和 subject，有助于保持提交的聚焦
3. **Header 精简、Body 详尽**：一句话说不清楚的事情放到 Body 里解释
4. **不兼容变更务必声明**：涉及接口变更时必须在 Footer 中写 BREAKING CHANGE
5. **及时提交**：不要积攒大量修改后一次性提交，小步快跑更易于回溯和定位问题
6. **善用 scope**：让 `git log --oneline` 的输出一目了然
